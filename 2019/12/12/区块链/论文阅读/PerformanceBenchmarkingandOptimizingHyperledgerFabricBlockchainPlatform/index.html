Performance Benchmarking and Optimizing Hyperledger Fabric Blockchain Platform

Posted on 2019-12-12 Edited on 2019-12-13 In 区块链

所做的优化

  • 加密操作是CPU密集的

    缓存Deserialization of identity. 把serialization 作为key,deserializtion的值作为value.
    Validation of identity with Organization’s MSP. hashmap来缓存每个identity对应于哪个organization。

  • 在一个块内并行验证block

    vscc阶段主要就是为了验证每个交易的背书
    可以配置每个peer在每个channel上的work数量，每个work负责一条消息的配置

  • Bulk Read/Write during MVCC Validation & Commit

    在MVCC阶段，对于CouchDB来说
    read set 中，对于每个交易的每一个key，都会调用一次GET REST API
    write set 中，对于每个交易的每一个key，都会调用一次GET REST API 取得version版本号，再调用PUT REST API
    显然这样调用非常耗时
    couchdb 建议用bulk operations

FastFabric: Scaling Hyperledger Fabric to 20,000 Transactions per Second

优化主要针对consensus模块

  1. 交易可能很大，在发给kafka排序时，只需要发txID即可排序，而不需要发送整个tx。可以很好地节省网络和内存开销。其他部分保存在本地
    然后等kafka返回结果，再来拼接相应部分。（这部分到底该怎么做呢？）
  2. ordering service 并行处理transaction
  3. 针对peer节点
    • 验证交易的读写集，使用in-memory hash table 来代替world state ,而不是database
    • block都是不可修改的，所以存储block很适合append-only。使用分布式存储来存储block,如 hadoop mapreduce or spark.
    • blockchain log可以推后写入，因为准确来讲，它不属于transaction flow.(但是如果宕机，这样真的安全吗？)
    • 有些peer节点既是commiter，又是endorser ，最好把它们分开（使用不同的硬件）
    • 验证state change必须是sequentially的
    • 并行验证 transaction
    • cache unmarshled blocks
