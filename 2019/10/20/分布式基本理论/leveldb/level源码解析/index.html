1.6 ValueType(db/dbformat.h)

levelDB更新某个key时，不会直接操作db中的数据，每次操作都是直接插入一份kv数据，具体的数据合并和清除由后台的compact完成。每次put，db中就会新插入一份kv数据，delete操作相当于Put空的value。使用ValueTpye来区分是否是删除数据。

1.7 SequenceNumber(db/dbformat.h)

leveldb的每次更新（put/delete)操作都拥有一个版本，由SequnceNumber来标识，整个db有一个全局值保存着当前使用的SequenceNumber。存储时，SequenceNumber只占了56 bits,ValueType占剩余的8bit

1.11 Comparator

InternalKeyComparator(db/dbformat.h)

db内部做key排序时使用的比较方法。排序时，会先使用user-comparator比较user-key,如果user-key相同，再比较SequenceNubmber的大小。因为SequenceNumber在db中全局递增，对于相同的user-key,最新的更新排在前面，在查找时会被先找到。

InternalKeyComparator 中 FindShortestSeparator()/ FindShortSuccessor()的实现，仅从传入的内部key参数，解析出user-key，然后再调用user-comparator的对应接口。

WriteBatch(db/write_batch.cc)

将多个操作（put/delete）封装成WriteBatch。

Memtable(db/memtable.cc/skiplist.h)

db数据在内存中的存储格式，写操作的数据会先写入到memtable中，memtable的实现是skiplist。
当memtable的大小达到一定阈值后，就会变成只读的imemtable。同时生成一个新的memtable来供写入。
后台运行的compact进程会负责将imemetable dump成新的sstable。

sstable(table/table.cc)

db数据持久化文件，提供只读。

block(table/block.cc)

sstable的数据由一个个的block组成。当持久化数据时，多份kv聚合成一个block写入；
读取数据时也是以block单位做IO。sstable的索引信息会保存符合key-range的block在文件中的offset/size(blockHandle)

BlockHandle(table/format.h)

block的元信息（位于sstable的offset/size)封装成BlockHandle.

Compaction(db/version_set.cc)

Version(db/version_set.cc)

将每次compact后的最新数据状态定义为Version，也就是当前db元信息以及每个 level上具有最新数据状态的sstable集合。

  • refs_

    compact 会在某个 level 上新加入或者删除一些sstable，但可能这个时候， 那些要删除的 sstable 正在被读，为了处理这样的读写竞争情况，基于 sstable 文件一旦生成就不会 改动的特点，每个Version加入引用计数，读以及解除读操作会将引用计数相应加减一。这样， db中可能有多个Version 同时存在(提供服务)，它们通过链表链接起来。当 Version的引用计数为 0并且不是当前最新的 Version 时，它会从链表中移除，对应的，该Version内的 sstable 就可以删除了 (这些废弃的sstable会在下一次compact完成时被清理掉)。

  • compaction_score

    • 在从sstable中查找数据时，每一层的sstable的key分布，以及size都会影响读取的效率。可以预想的最佳情形可能是level-0中最多有一个sstable，level-1 以及之上的各level中key-range分布均匀，期望更多的查找可以遍历最少的level即可定位到。
    • 这样的理想状态叫做：leveldb处于均衡状态。compaction_score_的分值越高，代表该level越不均衡，越应该优先进行compact。
    • 取每层的compaction_score_最大值作为该Version的compaction_score_，同时记录该层的compaction_level_

  • file_to_compact_

    • 如果对于一个sstable多次查找，则说明低层的sstable没有提供很好的hit率，倾向于认为compact可以提高均衡状态。但一个文件的寻道次数达到一定阈值（allowed_seeks)，就对其进行compact。

VersionSet(db/version_set.cc)

整个db的状态都由VersionSet管理着，其中包括当前Version的状态以及其他正在被使用的Version状态。

VersionEdit(db/version_edit.cc)

compact过程中会有一系列改变Version的操作，如 filenumber增加，删除sstalbe，为了缩小VersionSet的切换，将这些操作封装成VersionEdit,当compact完成时，将 VersionEdit 中的操作一次应用到当前 Version 即可得到最新状态的 Version。

VersionSet::Builder(db/version_set.cc)

辅助类，将所有VersionEdit中的操作应用到Version中。

Manifest(db/version_set.cc)

为了重启db后可以恢复推出前的状态，需要将db中的状态保存下来。这些信息就在manifest中。为了使db更快地恢复，manifest不会只保存当前的状态，而是将历史的状态都保存下来。又考虑到每次状态的完全保存需要的空间和耗费比较多。所以在开始时，manifest保存完整的状态信息（VersionSet::WriteSnapshot()),接下来只保存每次compact产生的操作（VersinEdit)，重启db时，根据开头的状态，依次将后续的VersionEdit replay，即可恢复到退出前的状态。

存储结构

前面提到了memtable中存储的是InternalKey

MemTable的Add操作：
主要是构造插入Mmetable中的数据，包括InternalKey和value，然后调用SkipList的Insert函数来插入数据。
排序按照userkey递增，sequenceNumber递减，这样可以保证相同userkey的快照排到一起，且最新的数据排在前面

思考： 在写入时，为什么不直接把原有的key替换呢，为了保证快照读吗？

MemTable的Get操作：

  • 构造查找
  • 调用Skiplist的Iterator。SkipList中的迭代器也就是调用FindGreaterOrEqual
  • 判断返回结果的user_key是否和查找key相等，如相等，判断ValueType:
    • KTypeValue,返回结果
    • KTypeDelete,返回 NotFound

SkipList

SkipList的数据结构

SkipList的插入操作：

  • 首先找到不小于当前key的插入位置，FindGreaterOrEqual函数返回在每个level层上的prev node.
  • 随机产生一个高度
  • 通过prev[i],在所有高度上插入该节点

FindGreaterOrEqual的大致过程：

  1. 从最高层开始，往后查找
  2. 如果在当前层找到插入位置，记录下前一个节点prev
  3. 在往下一层查找
  4. 循环第二步

Arena

为了便于内存使用，提高内存使用效率，以及更好地管理内存。level使用了Arena来管理内存。

分配内存也很简单

如果当前需要的内存小于剩余内存，直接将当前block的剩余内存分配。
否则就调用AllocateFallback申请新的内存。

Arena每次以kBlockSize（4096）为单位向系统申请内存。
需要注意的是调用AllocateFallback的前提是当前block已经没有足够的内存可以分配。

block of sstable (table/block_builder.cc table/block.cc)

数据在一个block中存储如下：

关于key做前缀压缩

  • 一个key/value形成一个entry
  • entry是key值经过前缀压缩存储的，即对于每一个key，只需要记录与前一key相同的长度，以及剩余不同部分的key即可。
  • 由于写入到block中的key/value都是由上层的memtable dump下来，或者是由上层的sstable merge的，所以key都是相对有效的 这样虽然有效地节省了存储空间，但是在查找key时，就需要从block的开头开始查找，非常低效。
  • 所以设置一个block_restart_interval，表示每轮压缩只包含这么多entry，超出就开始新的一轮压缩。restarts中记录了每一次开始新一轮压缩的位置，num_of_restarts记录了总的轮数。

压缩后key/value的存储

  • shared_bytes 表示与前一个key相同的长度
  • unshared_bytes 表示剩余不同的长度
  • value_bytes 表示value部分的长度
  • unshared_bytes 不同部分的key的值
  • value_data value部分的值

trailer

  • 每个 block 后面都会有 5 个字节的 trailer。
  • 1 个字节的 type 表示 block 内的数据是 否进行了压缩(比如使用了 snappy 压缩)，4 个字节的 crc 记录 block 数据的校验码。

BlockBuilder会负责加入key/value。

  1. 找到当前key和last_key相同的长度。
  2. 如果counter_小于block_restart_interval，那么继续在该轮中进行压缩，找到相同的key前缀。
  3. 如果counter_大于block_restart_interval：
    • 那么重新开始新的一轮，在restarts_记录下开始新一轮的位置；
    • counter_重置为0；
  4. 构造entry,写入到buffer_
  5. 更细last_key_

block的查询是封装在Iter类中
Iter中的Seek函数负责查找：二分查找

restarts_中记录了每一轮压缩的开始位置，通过二分查找，找到key所在的压缩区间，再查询该key是否在该区间中。

BlockHandle指向一个block，BlockHandle相当于block在sstalbe中的索引信息。

  • 拥有两个参数：offset_和size_
  • offset指该block在sstable中的位置
  • size指block大小，

tableHandle

  1. 如果是新开的block(pending_index_entry为true),那么需要加入index_entry。key为last_key,value为该block的offerset/size。注意这里的key不是直接去上一个block的最后一个key，而是找到最短的一个字符串，能区分开上一个block的最后一个key，和当前block的第一个key即可，这样可以很好地节省内存开销，因为index_block为load到内存里面。
