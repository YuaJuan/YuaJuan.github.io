The google File System 论文理解

Posted on 2019-10-10 Edited on 2019-10-13 In 分布式基本理论

1. 设计上的考虑

  • 1.1 组件失败是常见的
  • 1.2 文件很大

    文件很大且都包含很很多application的数据。直接管理成百上> 千kb大小的文件并不明知。所以采用block的方式来管理。

  • 1.3 多数应用场景是append 数据，而不是重写已有的数据
  • 1.4合作设计application以及file system api有利于整个系统

    为了简化系统牺牲了一定的一致性。
    设计了原子的append操作所以application可以同时进行append.

2.设计概述

2.1 Assumptions

  • 系统由一些便宜的组件构成，这些组件容易出现failure.所以必须持续地监控，detect,容错，恢复。
  • Multi-GB files是常见的情况，应该被高效地管理。小文件也支持，但是不做优化。
  • read主要有两种：
    • large streaming reads
    • small random reads
  • 大量序列的append数据到文件的写操作,小量任意的写操作也支持，但不提供优化。
  • 高效处理client同时append数据到同一文件。
  • 高带宽比低延迟更加重要。

2.2 Interface

支持常见的create,delete,open,close,read,wrtie files.还支持snapshot,以及record append操作。

2.3 架构设计

  • 一个master和多个chunkServers.
  • master维护所有系统文件的元数据。
  • 为了可靠性，一个chunk会复制多份，默认复制3份。
  • master用心跳来与server交流，收集他们的状态。
  • client通过与master通信，获取元数据。和数据相关的操作，client则直接与chunkServers通信。
  • 文件被分成固定大小的chunk来存储。

    client是chunkserver都不需要缓存数据

2.4 Single Master

只有一个master大大地简化了设计。

client从master那儿得到应该到哪个chunkserver上获取数据，然后client直接与chunkserver交互。

注意需要最小化master的读写，避免master成为性能瓶颈。

  1. 使用固定大小的chunk size。client将文件名以及该chunk在该文件中的字节偏移转化成chunk index发送给master。
  2. master接收到后，返回给client相应的chunck handler以及它的位置。
  3. client缓存下该信息，使用file name以及chunk index作为key.
  4. client发送给离自己最近的replicas。（后面读取该chunk也不需要和master交互，除非相应的数据过期）

2.5 Chunk Size的选择

GFS选择64MB作为chunk size大小，比一般的都大，原因如下：

  1. 减少client与master的交互，因为对同一chunk的读写，只需要一次reqeust.(因为client只需要知道复制该chunk的chunkserver的位置即可) 加上读写通常是连续的，即对同一chunk，所以非常有用。
  2. chunk更大，那么client可以在一个chunk上做更多的操作，这样client和chunkserver可建立持久化连接，来减少网络开销。
  3. master可以存储更少的元信息，那么master可以元信息存储在in-memory中。

2.6 元信息

master存储三个元信息：

  • the file and chunk namespaces
  • the mapping from files to chunks
  • the locations of each chunk’s replicas

前面两个信息通过logging的方式来持久化。最后一个信息不需要，因为master启动后可以和chunkserver交互来再次获得。

2.6.1 In-Memory Data Structures

master的元信息数据都在In-Memory中，所以非常高效。
且master可以很容易高效地周期性地scan所有信息来完成垃圾回收，re-replication，当有chunckserver fail时来完成负载均衡等。

2.6.2 Chunk Locations

master无需持久化chunk’s replicas的位置，因为它可以通过心跳机制来获取它们的位置。
(是不是在心跳信息中，也告知master chunk是否存入磁盘中)

2.6.3 Operation Log

Operation log 记录了元数据改变的历史操作。为了避免丢失数据，会在本地和远程都备份。当备份到本地和远程后才会返回结果给client(这样不会很低效吗！)

为了使master快速启动，需要设置checkpoint。这样大大减少了日志量，master只需要replay checkpoint之后的少量日志即可。

checkpint是通过将日志压缩为B-tree,启动后直接载入内存，即可进行namespace的查找，无需另外的parsing

创建checkpoint是需要时间的，怎么做到在创建checkpoint时，client是不会感知到延迟的呢？

在准备创建checkpoint时，会先创建新的log file，然后该新log file继续服务，旧的log file在另外的线程中进行处理来创建checkpoint.

2.7 一致性模型

为了高效和简化，GFS使用了弱一致性。

  • File namespace的改变是原子的
  • 在一系列成功的改变后，GFS接下来会：
    • 以相同的order应用在所有其他replica的chunk上
    • 使用chunk version 去检测是否有replica过期
      （一个stale的replica不再会做任何更改，且不会返回自己的位置当client向master请求replica的位置时）

      需要注意的是，client缓存了chunk locations。所以在chunk过期前，client还是可能于一个stale的chunk交互）

  • 在成功更改后，组件failuer仍然可能导致数据毁坏。GFS一旦检测到问题出现，就会马上从正常的replica中恢复数据。

3 系统 interactions

系统的设计考虑所有操作都减少master的参与。

3.1 Leases and Mutation Order

mutation操作是指改变元信息或者是改变chunk内容。
master会授予某个replica某些chunk的租期（60s)，该replica就被成为primary。primary决定某个顺序来执行对chunk的修改，其他replica根据相同的顺序来执行。

master通过心跳来授予租期，primary在完成mutation后可以继续向master申请租期。master也可以在租期过期前，重新分配租期。（如果master想要组织一个mutations，比如file的名字被修改）

采用数据流和控制流分离的方法。

具体的过程如下：

  1. client向master询问哪个chunkserver有当前的租期以及其他replica的位置。如果当前没有chunkserver持有租期，那么master会授予一个。

  2. master回复primary的identity以及其他replica的位置。客户端会缓存这些数据。只有当primary不可达或租期过期，client才需要再次请求master。

  3. 客户端会把数据push到所有的replicas.传输的方式如下：

    如果需要push数据给S1-S4,client首先把数据发给离自己最近的replica S1,S1再发给离自己最近的S2,依次类推。

    每个chunkserver都会缓存数据到内部的LRU.

  4. 所有replicas都有数据后，client发送write指令给primary.primary会给所有mutations分配serial number，然后应用到自己的state.

  5. primary会转发write request给所有其他的replica,安装serial number来应用到自己的state.

  6. replica完成后会回复primary,自己已经完成了操作。

  7. primary回复client,如何有任何error,也返回给client.(如果错误发生在primary，那么primary将不再分配serial number以及转发)

3.3 Atomic Record Appends

5.1 高可用性

主要有两个方法来保证高可用性：Fast recovery ，replication

5.1.1 Fast Recovery

master和chunkserver都设计为存储状态，能在几秒内恢复。

5.1.2 Chunk Replication

每个chunk都会复制到多个chunkserver上，且这些chunkserver在不同的机架上。

5.1.2 Master Replication

对master的任何修改需要先写入日志，才会执行。任何一个master进程fail 都可以快速重启。
“shadow” master 提供只读，可以稍微落后于真正的master

5.2 Data Integrity

GFS的修改，特别是前面讨论的原子记录附加，并不保证相同的副本。因此，每个chunkserver必须通过维护校验和独立地验证其副本的完整性。

每个chunk被分为64KB的blocks,每个都有一个对应的32位的checksum。
checksum和其他metadata一样，会在memory中存储，同样也会通过log的方式持久化到硬盘。

  • 对于read操作，chunkserver会验证对应data block的checksum ，然后再返回数据。
  • 如果发现有错误，chunserver会返回错误给requestor，并且报告reports mismatch给master。
  • requestor收到错误后，会再向其他replica发送请求，master将会从其他replica中克隆chunk。
