分布式共识算法：PBFT

Posted on 2019-09-13 Edited on 2019-02-21 In 分布式 , 分布式共识算法

什么是拜占庭问题

拜占庭问题：
拜占庭将军问题（Byzantine Generals Problem），是由莱斯利·兰波特在其同名论文中提出的分布式对等网络通信容错问题。拜占庭帝国是一个古老而强大的帝国，这次他们想攻打另一个强大的敌人，所以派出了10支军队想要包围敌人。并且这10支军队约定，互相依靠轻骑兵通信协商，在包围敌人之后同时进攻。那么问题来了，如果这其中有人叛变，传递错误的信息，那么怎么同时进攻来取得胜利呢？

在分布式计算中，不同的计算机通过通讯交换信息达成共识而按照同一套协作策略行动。但有時候，系统中的成员计算机可能出错而发送错误的信息，用于传递信息的通讯网络也可能导致信息损坏，使得网络中不同的成员关于全体协作的策略得出不同结论，从而破坏系统一致性。拜占庭将军问题被认为是容错性问题中最难的问题类型之一。

提出背景

恶意攻击和软件错误越来越常见。工业和政府对在线信息服务的日益依赖，使得恶意攻击更具吸引力，成功的攻击的后果更严重。此外，由于软件的规模和复杂性的增长，软件错误的数量也在增加。所以考虑容错的分布式共识系统尤为重要。

PBFT算法简介

算法在保证活性和安全性（liveness & safety）的前提下提供了(n-1)/3的容错性。
总节点数为N个。
假设有f个faulty副本节点，那么非faulty节点数为N-f个节点。这里假设这N-f个节点中可能有f个可能由于网络延迟等原因不能及时发送或收到消息。要求响应的正常节点数大于faulty节点数，则N-f-f>f。所以N的数目至少为3f+1。

PBFT算法前提，采用密码学算法保证节点之间的消息传送是不可篡改的。
作者使用了加密技术来防止欺骗攻击和重播攻击，以及检测被破坏的消息。消息包含了公钥签名（其实就是RSA算法）、消息验证编码（MAC）和无碰撞哈希函数生成的消息摘要（message digest）。使用m表示消息，mi表示由节点i签名的消息，D(m)表示消息m的摘要。按照惯例，只对消息的摘要签名，并且附在消息文本的后面。并且假设所有的节点都知道其他节点的公钥以进行签名验证。

算法流程

PBFT是一种状态机副本复制算法，所有的副本在一个视图（view）轮换的过程中操作，主节点通过视图编号以及节点数集合来确定，即：主节点 p = v mod |R|。v：视图编号，|R|节点个数，p：主节点编号。
PBFT算法大致步骤如下：

  1. 客户端向主节点发送请求调用服务操作
  2. 主节点通过广播将请求发送给其他副本
  3. 所有副本都执行请求并将结果发回到客户端
  4. 客户端需要等待f+1个不同副本节点发回相同的结果

    request阶段

客户端c向主节点发送<REQUEST,o,t,c>请求执行状态机操作o,这里时间戳t用了保证客户端请求只会被执行一次。

当主节点收到客户端的请求m，主节点将该请求发送给所有副本节点。

接下来就是three-phase protocol

  • 预准备pre-prepare
  • 准备 prepare
  • 确认 commit

预准备阶段(pre-prepare)

在预准备节点，主节点分配一个序列号n给收到的请求，然后向所有备份节点广播预准备消息，预准备消息的格式为<<PRE-PREPARE,v,n,d>,m>,这里v是视图编号，m是客户端发送的请求消息，d是请求消息的摘要。

请求本身不包含在pre-prepare消息中（只有请求消息），这样可以使得pre-prepare消息足够小。pre-prepare的目的是作为一种证明，确定该请求在视图v中被赋予了序号n，从而如果发生视图变更可以追索。另外一个层面，将“请求排序协议”和“请求传输协议”进行解耦（？？？？what，为什么这样就排序了，在哪个阶段进了请求传输？？？），有利于对消息传输的效率进行深度优化。

只有满足以下条件，各个备份节点才会接受一个预准备消息：

  • 请求和预准备消息的签名正确，并且d与m的摘要一致
  • 当前视图编号是v
  • 该备份节点从未在视图v中接受过序号为n但是摘要d不同的消息m
  • 预准备消息的序号n必须在水线上下限h和H之间
    (水线存在的意义在于防止一个失效节点使用一个很大的序号消耗序号空间)

准备(prepare)阶段

如果备份节点i接受了预准备消息<<PRE-PREPARE,v,n,d>,m>则进入到准备阶段。在准备阶段的同时，该节点向其他所有副本节点广播准备消息< prepare,v,n,d,i>,并且将预准备消息和准备消息写入自己的消息日志。
如果节点i没有接受预准备消息，则什么也不需要干

只有满足以下条件，各个备份节点才会接受一个准备消息：

  • 请求和预准备消息的签名正确
  • 当前视图编号是v
  • 预准备消息的序号n必须在水线上下限h和H之间

准备阶段完成的标志
准备阶段完成的标志位副本节点i将(m,v,n,i)记入到其消息日志，以及收到2f个从不同副本节点收到的与预准备消息一致的准备消息（加上自己就是m+1了）

  1. 其中m是请求内容
  2. 视图v
  3. 预准备消息m在视图v中的编号

确认(commit)阶段

准备阶段完成（即将(m,v,n,i)记入到其消息日志，以及收到2f个从不同副本节点收到的与预准备消息一致的准备消息），就向全网发送commit消息，< COMMIT,v,n,D(m),i>消息。
只有满足以下条件，各个备份节点才会接受一个commit消息：

  • 请求和预准备消息的签名正确
  • 当前视图编号是v
  • 预准备消息的序号n必须在水线上下限h和H之间

确认阶段完成的标志

  • committed(m,v,n)为真的条件：if and only if prepared(m,v,n,i)is ture for all i in some set of f+1 non-faulty replicas;
  • committed-local(m,v,n,i)为真的条件：if and only if prepared(m,v,n,i)is true and i has accepted 2f+1 commits(possibly including its own )from different replicas that match the pre-prepare for m;

a commit matches a pre-prepare if they have the same view, sequence number, and digest。

可以这么理解：committed-local是对于某一个节点来说可以做判读的，即自己的（m,v,n,i）为true，并且还收到了和预准备消息一致的2f+1个commits。committed要求f+1个正常节点的prepared(m,v,n,i)为true，没有办法直接判读。

非常重要的：对于任意正常节点i来说，如果committed-local为真，那么committed就为真。

这样能保证任何正常节点的本地确认最终会确认f+1个更多的正常副本。（应该是2f+1中有f个非正常副本吗？非正常副本和正常副本的区别是是什么？）

reply阶段

每个副本节点在commited-local(m,v,n,i)为真后，开始执行m的请求。每个节点在完成请求的操作后，都会向客户端发送执行结果。
副本节点会把时间戳比自己已经回复的更小的请求丢弃，保证请求只会被执行一次。

客户端等待f+1个从不同副本得到的同样的响应，同样的响应需要保证签名正确，并且具有同样的时间戳t和执行结果r。这样客户端才能把r当做正确的执行结果。

如果客户端没有在有效时间内收到回复，将会把请求向所有副本节点进行广播。如果请求已经在副本节点处理过了直接把结果返回给客户端，如果没有则会把请求转发给主节点。
如果主节点没有将请求进行广播，那么就认为主节点失效，如果有足够多的副本节点认为主节点失效，则会触发视图变更。

总结

  • 预准备阶段：<<PRE-PREPARE,v,n,d>,m>，主节点将消息在视图v中编号。（这个阶段只有自己收到了请求消息，但并不知道其他节点的情况）
  • 准备阶段：<<PRE-PREPARE,v,n,d>,m>，接受了预准备消息的副本节点广播准备消息。收到2f个从不同副本节点收到的与预准备消息一致的准备消息才认为准备阶段完成。（这个阶段能知道我收到的这个请求至少有另外2f个节点也收到了）
  • commit阶段：< COMMIT,v,n,D(m),i>，准备阶段完成的节点开始广播commit消息，prepared(m,v,n,i)为真，且收到2f+1个与预准备一致的commit。
  • reply阶段：commited-local(m,v,n,i)为真后，开始执行m的请求。每个节点在完成请求的操作后，都会向客户端发送执行结果。

检查点协议

在PBTF算法中，logs中存储了大量的预准备，准备和确认消息，为了节省内存，系统需要一种将日志中国的无异议消息记录删除的机制。

无异议消息：无异议消息记录是指副本节点在删除自己的消息日志之前，已经有f+1个正常副本节点执行了消息对应的请求，并且可以在视图变更时向其他副本节点证明。
但是如果在每一个操作执行后都生成这样的证明是非常消耗资源的，因此，证明过程只有在请求序号可以被某个常数（比如100)整除的时候才会周期性地执行。

将这些请求执行后得到的状态称作检查点(checkpoint),并且将具有证明的检查点称为稳定检查点(stable checkpoint)

检查点的正确性生成过程：

  1. 副本节点i生成检查点
  2. 副本节点i向其他副本节点广播检查点消息< CHECKPOINT,n,d,i>。这里n是最近一个影响状态的请求序号，d是状态摘要
  3. 每个副本节点在各自的日志中收集并记录其他节点发过来的检查点消息，直到收到来自2f+1个不同副本节点的具有相同需要n和摘要d的检查点消息。

这2f+1个消息就是这个检查点的正确性证明。

具有证明的检查点成为稳定检查点，然后副本节点就可以将所有序号小于n的预准备消息，准备消息和确认消息从日志中删除，同时也可以将之前的检查点和检查点消息一并删除。
