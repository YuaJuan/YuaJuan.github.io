IPFS - Content Addressed, Versioned, P2P File System

InterPlanetary File System(IPFS)是一个p2p的分布式文件系统，它将所有的电脑通过同一文件系统连接起来。
某种程度上将，IPFS类似Web，但是更像是BitTorrent swarm,在一个Git仓库中交换objects。
IPFS基于内容寻址的hyper links来提供一个高吞吐量的content-addressed block storage 模型。
IPFS形成一个Merkle DAG，这是一种可以构建versioned file systems，blockchains以及持久化web的数据结构。
IPFS结合分布式hashtable,激励模型的block exchange,self-certifying namespace。
IPFS没有单点失败，且nodes之间不需要去相信彼此。

Introduction

Napster, KaZaA, 和 BitTorrent 部署了大规模的分布式文件系统，但是这些具体的应用都不是提供一个基础架构来让开发者基于此，搭建上层应用。一方面可能是因为，大多数的需求场景已经有足够好的系统来提供服务了，例如“HTTP”。但是随着数据增多，“HTTP”架构可能不再适用了。

IPFS的原则就是：

The central IPFS principle is modeling all data as part of the same Merkle DAG.

背景知识

  • 分布式hash表

  • Block Exchanges

    受BitTorrent的影响，IPFS的设计包含如下：

    • 奖励做出贡献的节点，惩罚只获取资源的节点
    • 节点会track每个file pieces的availability，优先发送pieces最少的。

  • 版本控制系统-Git
    Git通过Merkle DAG来记录filesystem tree的修改。

  • Self-Certified Filesystems-SFS

IPFS DESIGN

IPFS是一个分布式文件系统，综合了之前的p2p系统的成功的idea，包括：DHTs,BItTorrent,Git,SFS。

IPFS协议分为如下几个子协议：

  • Identities
  • Network
  • Routing
  • Exchange
  • Object
  • Files
  • Naming

Identities

Network

  1. Transport: IPFS can use any transport protocol, and is best suited for WebRTC DataChannels (for browser connectivity) or uTP(LEDBAT [14]).
  2. Reliability: IPFS can provide reliability if underlying networks do not provide it, using uTP (LEDBAT [14]) or SCTP.
  3. Connectivity: IPFS also uses the ICE NAT traversal techniques.
  4. Integrity: optionally checks integrity of messages us- ing a hash checksum.
  5. Authenticity: optionally checks authenticity of mes- sages using HMAC with sender’s public key.

Note on Peer Address

IPFS可以使用任何网络，且不依赖于IP.IPFS可以在overlay网络中使用。IPFS使用multiaddr来提供地址给下层网络。multiaddr可以表达地址和协议，包括支持的封装。例如：

Routing

IPFS node需要一个Routing System能够找到其他peer的network地址并且能够获取服务于某个object的所有节点。IPFS基于S/Kademlia以及Coral使用DSHT。存储时，如果小value(不大于1KB)就直接存储在DHT中，如果大于1KB就存储该值的references。

Blokc Exchange-BitSwap Protocal

在IPFS中，peer使用基于BitTorrent的激励机制–bitSwap交换block，以达到来分发数据的目的。类似于BitTorrent,在BitSwap，peer需要的object在want_list中，并且可以分发的object在have_list中。但是在BitTorrent中有个限制，这些object都需要在一个torrent中，但是在BitSwap中没有这个限制。

BitSwap是一个持久的市场，节点可以在这里获取他们需要的数据块。由于barter系统的存在，那么隐含了虚拟currency可以被创建，这就需要一个全局的账本来追踪currency的owership和transfer.

有时候peers需要的object，该节点也没有。该节点也需要去向其他邻居获取。于此同时，该节点也有自己需要的object,但是这个的优先级会低于peers找该节点寻求的object，这样可以很好地激励节点缓存和广播稀少的pieces,即使他们本身对这些pieces并没有兴趣。

Version(db/version_set.cc)

将每次compact后的最新数据状态定义为Version，也就是当前db元信息以及每个 level上具有最新数据状态的sstable集合。

  • refs_

    compact 会在某个 level 上新加入或者删除一些sstable，但可能这个时候， 那些要删除的 sstable 正在被读，为了处理这样的读写竞争情况，基于 sstable 文件一旦生成就不会 改动的特点，每个Version加入引用计数，读以及解除读操作会将引用计数相应加减一。这样， db中可能有多个Version 同时存在(提供服务)，它们通过链表链接起来。当 Version的引用计数为 0并且不是当前最新的 Version 时，它会从链表中移除，对应的，该Version内的 sstable 就可以删除了 (这些废弃的sstable会在下一次compact完成时被清理掉)。

  • compaction_score

    • 在从sstable中查找数据时，每一层的sstable的key分布，以及size都会影响读取的效率。可以预想的最佳情形可能是level-0中最多有一个sstable，level-1 以及之上的各level中key-range分布均匀，期望更多的查找可以遍历最少的level即可定位到。
    • 这样的理想状态叫做：leveldb处于均衡状态。compaction_score_的分值越高，代表该level越不均衡，越应该优先进行compact。
    • 取每层的compaction_score_最大值作为该Version的compaction_score_，同时记录该层的compaction_level_

  • file_to_compact_

    • 如果对于一个sstable多次查找，则说明低层的sstable没有提供很好的hit率，倾向于认为compact可以提高均衡状态。但一个文件的寻道次数达到一定阈值（allowed_seeks)，就对其进行compact。

VersionSet(db/version_set.cc)

整个db的状态都由VersionSet管理着，其中包括当前Version的状态以及其他正在被使用的Version状态。

VersionEdit(db/version_edit.cc)

compact过程中会有一系列改变Version的操作，如 filenumber增加，删除sstalbe，为了缩小VersionSet的切换，将这些操作封装成VersionEdit,当compact完成时，将 VersionEdit 中的操作一次应用到当前 Version 即可得到最新状态的 Version。

VersionSet::Builder(db/version_set.cc)

辅助类，将所有VersionEdit中的操作应用到Version中。

Manifest(db/version_set.cc)

为了重启db后可以恢复推出前的状态，需要将db中的状态保存下来。这些信息就在manifest中。为了使db更快地恢复，manifest不会只保存当前的状态，而是将历史的状态都保存下来。又考虑到每次状态的完全保存需要的空间和耗费比较多。所以在开始时，manifest保存完整的状态信息（VersionSet::WriteSnapshot()),接下来只保存每次compact产生的操作（VersinEdit)，重启db时，根据开头的状态，依次将后续的VersionEdit replay，即可恢复到退出前的状态。

关于key做前缀压缩

  • 一个key/value形成一个entry
  • entry是key值经过前缀压缩存储的，即对于每一个key，只需要记录与前一key相同的长度，以及剩余不同部分的key即可。
  • 由于写入到block中的key/value都是由上层的memtable dump下来，或者是由上层的sstable merge的，所以key都是相对有效的 这样虽然有效地节省了存储空间，但是在查找key时，就需要从block的开头开始查找，非常低效。
  • 所以设置一个block_restart_interval，表示每轮压缩只包含这么多entry，超出就开始新的一轮压缩。restarts中记录了每一次开始新一轮压缩的位置，num_of_restarts记录了总的轮数。

压缩后key/value的存储

  • shared_bytes 表示与前一个key相同的长度
  • unshared_bytes 表示剩余不同的长度
  • value_bytes 表示value部分的长度
  • unshared_bytes 不同部分的key的值
  • value_data value部分的值

trailer

  • 每个 block 后面都会有 5 个字节的 trailer。
  • 1 个字节的 type 表示 block 内的数据是 否进行了压缩(比如使用了 snappy 压缩)，4 个字节的 crc 记录 block 数据的校验码。

BlockBuilder会负责加入key/value。

  1. 找到当前key和last_key相同的长度。
  2. 如果counter_小于block_restart_interval，那么继续在该轮中进行压缩，找到相同的key前缀。
  3. 如果counter_大于block_restart_interval：
    • 那么重新开始新的一轮，在restarts_记录下开始新一轮的位置；
    • counter_重置为0；
  4. 构造entry,写入到buffer_
  5. 更细last_key_

block的查询是封装在Iter类中
Iter中的Seek函数负责查找：二分查找

restarts_中记录了每一轮压缩的开始位置，通过二分查找，找到key所在的压缩区间，再查询该key是否在该区间中。

The google File System 论文理解

Posted on 2019-10-10 Edited on 2019-10-13 In 分布式基本理论

1. 设计上的考虑

  • 1.1 组件失败是常见的
  • 1.2 文件很大

    文件很大且都包含很很多application的数据。直接管理成百上> 千kb大小的文件并不明知。所以采用block的方式来管理。

  • 1.3 多数应用场景是append 数据，而不是重写已有的数据
  • 1.4合作设计application以及file system api有利于整个系统

    为了简化系统牺牲了一定的一致性。
    设计了原子的append操作所以application可以同时进行append.

2.设计概述

2.1 Assumptions

  • 系统由一些便宜的组件构成，这些组件容易出现failure.所以必须持续地监控，detect,容错，恢复。
  • Multi-GB files是常见的情况，应该被高效地管理。小文件也支持，但是不做优化。
  • read主要有两种：
    • large streaming reads
    • small random reads
  • 大量序列的append数据到文件的写操作,小量任意的写操作也支持，但不提供优化。
  • 高效处理client同时append数据到同一文件。
  • 高带宽比低延迟更加重要。

2.2 Interface

支持常见的create,delete,open,close,read,wrtie files.还支持snapshot,以及record append操作。

2.3 架构设计

  • 一个master和多个chunkServers.
  • master维护所有系统文件的元数据。
  • 为了可靠性，一个chunk会复制多份，默认复制3份。
  • master用心跳来与server交流，收集他们的状态。
  • client通过与master通信，获取元数据。和数据相关的操作，client则直接与chunkServers通信。
  • 文件被分成固定大小的chunk来存储。

    client是chunkserver都不需要缓存数据

2.4 Single Master

只有一个master大大地简化了设计。

client从master那儿得到应该到哪个chunkserver上获取数据，然后client直接与chunkserver交互。

注意需要最小化master的读写，避免master成为性能瓶颈。

  1. 使用固定大小的chunk size。client将文件名以及该chunk在该文件中的字节偏移转化成chunk index发送给master。
  2. master接收到后，返回给client相应的chunck handler以及它的位置。
  3. client缓存下该信息，使用file name以及chunk index作为key.
  4. client发送给离自己最近的replicas。（后面读取该chunk也不需要和master交互，除非相应的数据过期）

2.5 Chunk Size的选择

GFS选择64MB作为chunk size大小，比一般的都大，原因如下：

  1. 减少client与master的交互，因为对同一chunk的读写，只需要一次reqeust.(因为client只需要知道复制该chunk的chunkserver的位置即可) 加上读写通常是连续的，即对同一chunk，所以非常有用。
  2. chunk更大，那么client可以在一个chunk上做更多的操作，这样client和chunkserver可建立持久化连接，来减少网络开销。
  3. master可以存储更少的元信息，那么master可以元信息存储在in-memory中。

2.6 元信息

master存储三个元信息：

  • the file and chunk namespaces
  • the mapping from files to chunks
  • the locations of each chunk’s replicas

前面两个信息通过logging的方式来持久化。最后一个信息不需要，因为master启动后可以和chunkserver交互来再次获得。

2.6.1 In-Memory Data Structures

master的元信息数据都在In-Memory中，所以非常高效。
且master可以很容易高效地周期性地scan所有信息来完成垃圾回收，re-replication，当有chunckserver fail时来完成负载均衡等。

2.6.2 Chunk Locations

master无需持久化chunk’s replicas的位置，因为它可以通过心跳机制来获取它们的位置。
(是不是在心跳信息中，也告知master chunk是否存入磁盘中)

2.6.3 Operation Log

Operation log 记录了元数据改变的历史操作。为了避免丢失数据，会在本地和远程都备份。当备份到本地和远程后才会返回结果给client(这样不会很低效吗！)

为了使master快速启动，需要设置checkpoint。这样大大减少了日志量，master只需要replay checkpoint之后的少量日志即可。

checkpint是通过将日志压缩为B-tree,启动后直接载入内存，即可进行namespace的查找，无需另外的parsing

创建checkpoint是需要时间的，怎么做到在创建checkpoint时，client是不会感知到延迟的呢？

在准备创建checkpoint时，会先创建新的log file，然后该新log file继续服务，旧的log file在另外的线程中进行处理来创建checkpoint.

2.7 一致性模型

为了高效和简化，GFS使用了弱一致性。

  • File namespace的改变是原子的
  • 在一系列成功的改变后，GFS接下来会：
    • 以相同的order应用在所有其他replica的chunk上
    • 使用chunk version 去检测是否有replica过期
      （一个stale的replica不再会做任何更改，且不会返回自己的位置当client向master请求replica的位置时）

      需要注意的是，client缓存了chunk locations。所以在chunk过期前，client还是可能于一个stale的chunk交互）

  • 在成功更改后，组件failuer仍然可能导致数据毁坏。GFS一旦检测到问题出现，就会马上从正常的replica中恢复数据。

3 系统 interactions

系统的设计考虑所有操作都减少master的参与。

3.1 Leases and Mutation Order

mutation操作是指改变元信息或者是改变chunk内容。
master会授予某个replica某些chunk的租期（60s)，该replica就被成为primary。primary决定某个顺序来执行对chunk的修改，其他replica根据相同的顺序来执行。

master通过心跳来授予租期，primary在完成mutation后可以继续向master申请租期。master也可以在租期过期前，重新分配租期。（如果master想要组织一个mutations，比如file的名字被修改）

采用数据流和控制流分离的方法。

具体的过程如下：

  1. client向master询问哪个chunkserver有当前的租期以及其他replica的位置。如果当前没有chunkserver持有租期，那么master会授予一个。

  2. master回复primary的identity以及其他replica的位置。客户端会缓存这些数据。只有当primary不可达或租期过期，client才需要再次请求master。

  3. 客户端会把数据push到所有的replicas.传输的方式如下：

    如果需要push数据给S1-S4,client首先把数据发给离自己最近的replica S1,S1再发给离自己最近的S2,依次类推。

    每个chunkserver都会缓存数据到内部的LRU.

  4. 所有replicas都有数据后，client发送write指令给primary.primary会给所有mutations分配serial number，然后应用到自己的state.

  5. primary会转发write request给所有其他的replica,安装serial number来应用到自己的state.

  6. replica完成后会回复primary,自己已经完成了操作。

  7. primary回复client,如何有任何error,也返回给client.(如果错误发生在primary，那么primary将不再分配serial number以及转发)

3.3 Atomic Record Appends

5.1 高可用性

主要有两个方法来保证高可用性：Fast recovery ，replication

5.1.1 Fast Recovery

master和chunkserver都设计为存储状态，能在几秒内恢复。

5.1.2 Chunk Replication

每个chunk都会复制到多个chunkserver上，且这些chunkserver在不同的机架上。

5.1.2 Master Replication

对master的任何修改需要先写入日志，才会执行。任何一个master进程fail 都可以快速重启。
“shadow” master 提供只读，可以稍微落后于真正的master

5.2 Data Integrity

GFS的修改，特别是前面讨论的原子记录附加，并不保证相同的副本。因此，每个chunkserver必须通过维护校验和独立地验证其副本的完整性。

每个chunk被分为64KB的blocks,每个都有一个对应的32位的checksum。
checksum和其他metadata一样，会在memory中存储，同样也会通过log的方式持久化到硬盘。

  • 对于read操作，chunkserver会验证对应data block的checksum ，然后再返回数据。
  • 如果发现有错误，chunserver会返回错误给requestor，并且报告reports mismatch给master。
  • requestor收到错误后，会再向其他replica发送请求，master将会从其他replica中克隆chunk。

数据抽象是一种依赖于接口和实现分离的编程技术。类的接口包括用户所能执行的操作；类的实现则包括类的数据成员，负责接口实现的函数以及定义类所需的各种私有函数。

封装后的类隐藏了它的实现细节，类的用户只能使用接口而无法访问实现部分.
