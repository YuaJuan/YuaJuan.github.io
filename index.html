The google File System 论文理解

Posted on 2019-10-10 In 分布式基本理论

1. 设计上的考虑

  • 1.1 组件失败是常见的
  • 1.2 文件很大

    文件很大且都包含很很多application的数据。直接管理成百上> 千kb大小的文件并不明知。所以采用block的方式来管理。

  • 1.3 多数应用场景是append 数据，而不是重写已有的数据
  • 1.4合作设计application以及file system api有利于整个系统

    为了简化系统牺牲了一定的一致性。
    设计了原子的append操作所以application可以同时进行append.

2.设计概述

2.1 Assumptions

  • 系统由一些便宜的组件构成，这些组件容易出现failure.所以必须持续地监控，detect,容错，恢复。
  • Multi-GB files是常见的情况，应该被高效地管理。小文件也支持，但是不做优化。
  • read主要有两种：
    • large streaming reads
    • small random reads
  • 大量序列的append数据到文件的写操作,小量任意的写操作也支持，但不提供优化。
  • 高效处理client同时append数据到同一文件。
  • 高带宽比低延迟更加重要。

2.2 Interface

支持常见的create,delete,open,close,read,wrtie files.还支持snapshot,以及record append操作。

2.3 架构设计

  • 一个master和多个chunkServers.
  • master维护所有系统文件的元数据。
  • 为了可靠性，一个chunk会复制多份，默认复制3份。
  • master用心跳来与server交流，收集他们的状态。
  • client通过与master通信，获取元数据。和数据相关的操作，client则直接与chunkServers通信。
  • 文件被分成固定大小的chunk来存储。

    client是chunkserver都不需要缓存数据

只有一个master大大地简化了设计。

client从master那儿得到应该到哪个chunkserver上获取数据，然后client直接与chunkserver交互。

C++面向对象

Posted on 2019-09-21 In 编程基础

类的基本思想是数据抽象和封装。

数据抽象是一种依赖于接口和实现分离的编程技术。类的接口包括用户所能执行的操作；类的实现则包括类的数据成员，负责接口实现的函数以及定义类所需的各种私有函数。

封装后的类隐藏了它的实现细节，类的用户只能使用接口而无法访问实现部分.

类的声明

就像把可以把函数的声明和定义分开来一样，我们也可以仅声明类而不去定义它。

这种声明有时被称作前向声明。它向程序中引用了名字screen并且指明它是一种类类型。scree在声明之后定义之前属于不完全类型。
对于一个类类型来说，在创建它的对象之前必须要先定义它。因为之后定义之后，编译器才能知道这个类应该分配的空间大小。

但是类允许包含只想它自身类型的引用或者指针；

const成员函数

如上即是一个const成员函数。表示this是一个常量指针。
这样的函数可以访问访问所有成员变量，但不能修改他们的值。

友元

如果想要访问类的私有成员或私有成员函数，那么需要将访问它的类或者函数声明为友元。

友元的定义需要在类内部
一般来说在类的开头或者结尾来声明友元

比特币交易脚本及交易数据(一)

Posted on 2019-09-13 Edited on 2019-02-22 In 区块链 , 比特币原理

1. 比特币交易本质

  • 交易实质是包含了一组输入列表和输出列表的数据结构，也就是转账记录
  • inputs用了追溯上一笔交易，以便明确转出者是否有权动用这笔钱，Outputs用来进行一次新的加密，加密后只有收款者才能解密并动用这笔钱
  • 输入中包含解锁脚本(unlocking script),输出中包含锁定脚本(locking script)
  • 锁定脚本往往含有一个公钥或者比特币地址，常用scriptPubKey来表示
  • 解锁脚本往往是支付方式用自己的私钥所做的签名，被称为签名脚本，用scriptSig表示
  • 在一个交易中，锁定脚本相当于是加密难题，解锁脚本是解开锁定脚本的题解。但解锁脚本的解题是针对上一个交易输出产生的加密难题的题解。
