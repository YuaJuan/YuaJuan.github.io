首先从redis的数据结构看起，这部分代码可以单独来理解，redis中的主要数据结构有下面几种：

  • 动态字符串：sds.h 和 sds.c

  • 字典：dict.h 和 dict.c

  • HyperLogLog：hyperloglog.c 中的 hllhdr 结构， 以及所有以 hll 开头的函数

动态字符串(sds.h 和 sds.c)

redis里，C字符串只会被用作字符串的字面常量，即用在一些不需要对字符串值进行修改的地方。
在其他情况下，会用到自己实现的动态字符串（simple dynamic string,SDS)。
sds的一些特性：

  1. 可以进行长度控制
  2. 二进制安全，即字符中间可以有’\0’
  3. 自动进行扩容

sds的定义如下，这么做是为了兼容c字符串，使得sds也可以调用c字符串的api。

MakeRoomFor是如何扩充新长度的？
只有当newlen大于当前sdshdr的alloc长度时，才会进行扩展。
newlen是这次实际需要的字符串长度。
如果当前长度newlen小于SDS_MAX_PREALLOC (默认为1024*1024)，那么实际扩展出的长度为newlen的2倍；
否则的话实际扩展出的长度为newlen加上SDS_MAX_PREALLOC。

注意在重新分配空间时，这里用到的是zrealloc，那么它与zalloc的区别是什么呢

zrealloc函数原型是extern void *realloc(void *mem_address, unsigned int newsize)
首先会去判断mem_address指向的地址是否有足够newsize的空间，如果有，直接扩大原来mem_address即可，并将mem_address返回即可。如果原地址空间不够，那么重新分配newsize大小的空间，并把原来mem_address的内容一一复制到新地址，mem_address原来指向的地址会被释放掉，并返回新地址。

2.2 rehash

什么时候会进行rehash?
used/size大于或者小于某个值时，此时会进行rehash（扩展或者收缩）

size会变为多少?

  1. 如果是扩展操作，那么ht[1]的大小为第一个大于等于ht[0].used*2的2^n；
  2. 如果是收缩操作，那么ht[1]的大小为第一个大于等于ht[0].used的2^n；

加入新的entry

首先调用_dictKeyIndex方法获取该entry应该放入的位置，如果该key已经存在，那么返回DICT_ERR。
在插入的时候采用了头插法，即将新的这个entry放在table对应的位置的首地址。
采用头插法的好处：
数据的局部性，新插入的数据，近期被访问的概率更大，所以放在前面的地址更高效。

在_dictKeyIndex函数中：

  1. 如果发现used等于了size了，那么会触发扩容。

  2. 计算entry应该放置的位置，是根据key计算hash，和sizemask来计算。

    因为size是2的n次方，sizemask是size-1,那么sizemask每一位都是1。举个例子，如果size=16,那么sizemask=15,转>化为二进制即是1111,那么用key计算的hash值和sizemask来求与，保证每一位都参与计算。

判断是否需要扩容： 如果used等于了size了则触发扩容，所以可以看出默认的负载因子就是1.
dictExpand并不是做真正的rehash，它只会扩大内存，真正的rehash是在访问时，才进行的，这叫做渐进式rehash。
那么其他操作是如何知道当前如果rehash呢？rehashidx不是-1时，就应该进行rehash。

因为在rehash时，该线程就无法做其他操作，所以每次rehash是有限制的，如果遇到非空的slot,那么将这个slot进行rehash，如果遇到连续的空slot,最多访问10个空slot就会返回。
在最后如果发现原谅的dict已经全部被rehash了，那么释放掉原谅ht[0]的空间，执行d->ht[0] = d->ht[1]。

HyperLogLog

HyperLogLog是redis中用来做基数统计的。

什么是基数

即一组元素中不重复的所有元素，比如一组元素{1,2,2,3,4,5,5,5,6,7,7},不重复的数为{1,2,3,4,5,6,7},所以基数为7。

hyperLog用法

  • PFADD key element [element …] 添加指定元素到 HyperLogLog 中。
  • PFCOUNT key [key …] 返回给定 HyperLogLog 的基数估算值。
  • PFMERGE destkey sourcekey [sourcekey …] 将多个 HyperLogLog 合并为一个 HyperLogLog

hyperLog原理

redis中HyperLog实现原理

将值通过hash后得到64位的值，用前14位来计算桶位，2的14次方是16384，所以需要16384个桶位。剩下的50位的第一个1出现的位置，需要装入桶中，最大的值就是50，需要用6个bit来表示。理论上需要的内存大小16384*6=12k。

编码方式

这里编码方式其实就是指redis中如何去实现上面讲到的16384*6。
redis中针对hyperLogLog有两种编码方式：稠密编码和稀疏编码

  1. 稠密编码
    稠密编码很容易理解：每6bit表达一个桶位即可。
  2. 稀疏编码
    虽然12k大小已经很小了，但redis对内存的优化是非常严格的。稀疏编码有3种表达：
  • ZERO: 00xxxxxx, 占用一个byte，前2位的0表示ZERO表达，后面6位表示连续的0的桶位的个数
  • XZERO: 01xxxxxx xxxxxxxx,占用两个byte,这个可以理解位ZERO的增强版，因为上面一共只有6位来表达，最大能表达64。XZERO用14位来表达,可最大表达16384
  • VAL: 1vvvvvxx,占用一个byte，中间的5位来表示具体的值，后面的2位来表示前面的值出现的次数，所以可以表达的最大的数为32，重复次数1-4.

从上面可以看出，稀疏编码最大能表达的数是32，所以当有值超过32时，就不得不转为稠密编码。

稀疏编码的例子：假如在桶位为1001，1002，1010的值分别为2，2，3，其余桶位均为0，那么稀疏编码表达如下：
1001，1002，1010的二进制分别为，11 1110 1001，11 1111 0010

XZERO: 0100 0011 1110 1000 表示0-1000都为0
VAL: 1 00010 02 表示1001，1002连续出现两个2
ZERO: 0000 0111 表示1003-1009连续7个0
VAL: 1 00011 01 表示1010是3
XZERO: 0111 1100 0000 1110 表示从1011到16384都为0

这里一共用了2+1+1+1+2=7个字节，相较于稠密编码用到了12k节约了很大的内存。
